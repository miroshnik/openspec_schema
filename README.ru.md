# spec-first — схема OpenSpec

> 🇬🇧 Английская версия: [README.md](README.md)

**Принцип #0 — spec есть единственный источник правды.** Код, тесты, проверки, миграции
и документация — всё это *проекции* спецификаций: тест проецирует сценарий — сценарий
(Given/When/Then, или conforms/violates) есть каноническое, СТЕК-АГНОСТИЧНОЕ утверждение свойства,
а тест — его локальный, перепроецируемый артефакт (когда сценарий меняется, ПЕРЕПРОЕЦИРУЙТЕ его тест).
ФОРМА проекции зависит от стека — unit / e2e / property / contract / type-level / рантайм-ассерт-тест
ИЛИ lint/AST-правило — что бы этот стек ни использовал для проверки ТОГО сценария. Миграция проецирует
изменение, код рождается в `apply` из tasks (которые проецируются из спецификаций). При любом расхождении
spec обладает приоритетом и меняется ПЕРВЫМ — вы никогда не правите производное за спиной его спецификации.
Всё остальное в этом README — две формы спецификации, CORE во frontmatter, порядок воркфлоу,
пер-спецификационный `test:`, gate, управление «всё-через-воркфлоу» — существует, чтобы насадить это
единственное правило.

> **Воркфлоу OpenSpec с проверками, которым можно доверить вашу генерацию, плюс стандарты,
> обязательные во всём проекте — насаждаемые, а не подшитые в вики.**

## Проблема

ИИ-агенты быстро генерируют код — но доверять можно лишь тому, что вы способны **проверить**, а правила,
которые на самом деле определяют кодовую базу (как в ней устроены auth, ошибки, логирование, слоистость),
живут в головах людей или в вики, которую никто не читает. **spec-first** делает и то, и другое
полноправными гражданами:

- **Функциональное поведение** *и* **сквозные стандарты** живут как спецификации под `specs/`.
- **Каждая спецификация несёт обязательный `test:`** — единственную ссылку на насаждение: каждый
  сценарий проецируется в тест, чья ФОРМА — выбор стека (unit / e2e / property / contract /
  type-level / рантайм-ассерт-тест ИЛИ lint/AST-правило) — поведенческий сценарий обычно в исполняемый
  тест, чисто структурное правило МОЖЕТ проецироваться в lint/AST-проверку.
- Единый **gate** должен быть зелёным до завершения изменения (полный набор тестов + целостность org +
  покрытие + регрессия + `validate --strict` + judge).
- То, что не может проверить ни одна машина, рецензирует **LLM-судья (judge)** — жёстко локально, мягко в CI
  (он никогда не краснит общую сборку).
- **Стандарт** — это сквозное правило, которое связывает каждую поверхность под его управлением во *всём
  проекте* — насаждаемое gate, а не подшитое в вики. (Опционально его можно сделать общим и между
  проектами — это бонус, ниже — но суть не в этом.)

Внедрение инкрементально: spec-first управляет **изменениями**, а не вашим существующим деревом — gate
храпово поднимает покрытие по одному изменению за раз и остаётся зелёным на всём пути. Это **community-схема**
— оверлей, который вы форкаете в свой проект (построенный на OpenSpec-овском `spec-driven`), так что
она наследует формат delta-спецификаций, на который опираются `archive` / `sync`.

## Что gate доказывает, а что нет

Вот честная формулировка, и в ней весь смысл дизайна:

**Зелёный означает**, что spec, тест и код СОГЛАСОВАНЫ, и что ни один сценарий молча не регрессировал.
Он делает spec тем, что вы *рецензируете*. Он НЕ доказывает, что spec корректна или что её тест
силён: один и тот же агент может написать spec, сценарий, проверку и тест, поэтому пустотелая
самосогласованная спецификация тоже станет зелёной. Спецификация становится доверенной лишь тогда, когда человек
её **ратифицирует** на ревью PR. Gate делает расхождение-со-спецификацией механическим; **ратификация**
делает саму спецификацию авторитетной.

Итак, разделение доверия таково:

| Что | Кто/что это гарантирует |
| --- | --- |
| Проекции согласованы со спецификацией | gate (механически) |
| Ни один сценарий не исчез молча | gate (diff регрессии) |
| Каждый сценарий действительно спроецирован | gate (покрытие) |
| Немеханизируемая часть соответствует духу | judge (выводит на поверхность; человек ратифицирует) |
| **Сама спецификация истинна / её тесты сильны** | **человек, при одобрении PR** |

Покрытие доказывает, что каждый СЦЕНАРИЙ ПРИВЯЗАН к помеченному, подключённому тесту, который ВЫПОЛНЯЕТСЯ —
оно не может доказать, что тест добросовестно или нетривиально прорабатывает сценарий (пустой `it()` с
правильным маркером `@spec` проходит). Это привязка, а не сила. *Сила* теста, как и *истинность*
спецификации — это то, что проверяет ратификация на ревью PR.

## Модель — две формы, один механизм

Один механизм, **две формы спецификации**, различаемые единственным полем frontmatter, `governs:`:

- **Функциональная capability** — поведение компонента. Тело: типизированный интерфейс (provides/requires)
  + поведенческие сценарии (Given/When/Then). Клаузулы `### Requirement:` SHALL/MUST И ЕСТЬ авторитетные
  свойства (межсценарное свойство, если оно именуется, живёт в прозе — не во frontmatter).
  **Её сценарии становятся её тестами.** Без `governs:`.
- **Стандарт** (`governs:` присутствует) — сквозное правило, которому подчиняется весь проект (соглашение
  по коду, планка безопасности/производительности/наблюдаемости, архитектурное решение). Это *обычная*
  спецификация под `specs/` — тот же delta-формат, тот же воркфлоу — сделанная сквозной благодаря `governs:`. Тело:
  правило как настоящее `### Requirement:` (SHALL/MUST + сценарий **conforms** И сценарий **violates**)
  + примеры conforms/violates (которые служат заодно тестовыми фикстурами).
  Архитектурные стандарты дополнительно несут полный **ADR** (`## Decision Record`: Status /
  Context / Decision / Consequences / Alternatives) плюс завершающий `## Rationale log`, который
  накапливается через ревизии. **Её сценарии conforms/violates, спроецированные в тест (mechanical),
  или зарегистрированный прогон judge (judge) — это её `test:`** — конкретный насаждающий механизм
  (unit/contract/type/runtime-тест ИЛИ lint/AST-правило, что бы стек ни использовал для проверки
  правила) назван в теле теста и в прозе, а не во frontmatter.

Обе формы подчиняются ОДНОМУ жёсткому контракту OpenSpec, который есть у каждой спецификации: `## Purpose` + `## Requirements`,
≥1 требование, несущее литеральные `SHALL`/`MUST`, ≥1 `#### Scenario:`.

### CORE во frontmatter

Обе формы несут один и тот же машиночитаемый CORE во **frontmatter** (выше `## Purpose`, чтобы archive
его сохранял; фиксированное место, чтобы gate и извлечение в design находили его вне зависимости от формы тела):

```
test:       <ссылка на тест(ы), которые проецируют сценарии этой спецификации — ключ покрытия>  # ОБЯЗАТЕЛЬНО
tier:       mechanical | judge          # машиночекаемо vs нужно-суждение; ВЕРИФИЦИРУЕТСЯ GATE'ом против того, во что разрешается test:
governs:    <scope>                     # только для стандартов; ПРИСУТСТВИЕ помечает стандарт, ОТСУТСТВИЕ ⇒ функциональная capability
adoption:   legacy                      # ОПЦИОНАЛЬНО — присутствие освобождает ранее существовавшую спецификацию от покрытия до первого изменения; ОПУСТИТЕ на активных спецификациях
```

Всего четыре поля. `test:` — ЕДИНСТВЕННАЯ ссылка на насаждение, ключ покрытия, который разрешает gate; для
`standard@mechanical` конкретный насаждающий механизм (unit/contract/type/runtime-тест ИЛИ
id lint/AST-правила, как `eslint-plugin-local/require-rate-limit` — выбор стека) назван в теле теста
и в прозе спецификации, а НЕ как поле frontmatter. Нет `check:` (свёрнут в `test:`), нет `invariant:` (клаузула `### Requirement:`
SHALL/MUST И ЕСТЬ авторитетный one-liner) и нет `binding:` (выводится). `governs:` — это
дискриминатор: его ПРИСУТСТВИЕ помечает стандарт; ОТСУТСТВИЕ ⇒ функциональная capability. `adoption:`
теперь — присутствие-`legacy` (опустите ⇒ active), а не enum.

Это **соглашения** spec-first, а не нативные поля OpenSpec — `validate`/`archive` их никогда
не читают (проверено на 1.4.1: спецификация, несущая CORE, возвращает `valid:true`, 0 проблем под
`--strict`). Убедитесь, что ваш форкнутый `validate` трактует неизвестный frontmatter как игнорируемый,
иначе gate покраснеет на `validate --strict` для каждой спецификации.

**`tier` ВЕРИФИЦИРУЕТСЯ GATE'ом, `binding` ВЫВОДИТСЯ — ни то, ни другое не принимается на веру.** `tier`
— это машиночекаемо-в-принципе (mechanical) против нужно-человеческое-суждение (judge), это НЕ механизм;
ПРИРОДА сценария подсказывает КЛАСС проекции (поведенческий → исполняемый тест; структурный → статический
анализ), но конкретный инструмент зависит от стека. Gate утверждает, что объявленный `tier` совпадает с
тем, во что разрешается `test:` (машиночекаемый тест-артефакт ⇒ mechanical; зарегистрированный прогон
judge ⇒ judge), и ПАДАЕТ при несовпадении — та же дисциплина, что и «binding выводится». Клаузула связывает
ТОГДА И ТОЛЬКО ТОГДА, когда её `test:` разрешается; поля `binding:` нет, проверка покрытия операционализирует это.

### Уровни — два, без лазеек

- **mechanical** — AST / lint / type / test. Уровень жёсткой связки. Механизируйте в первую очередь.
- **judge** — сфокусированный per-standard LLM-обзор той части, которую машина поймать не может (намерение /
  именование / эргономическая уместность). ОБЯЗАН ВЫПОЛНЯТЬСЯ и ВЫВОДИТЬ НА ПОВЕРХНОСТЬ (его нельзя пропустить), но он НЕ
  самонасаждается: **жёстко локально** (блокирует закрытие задачи), **мягко в CI** (постит подозреваемых),
  СВЯЗЫВАЕТ лишь когда человек ратифицирует его подозреваемых на ревью PR. Каждая judge-клаузула ОБЯЗАНА
  зарегистрировать запускаемую judge-проверку.

Advisory-уровня нет. Каждое правило — это ПАРА: авторская guidance, которой следует агент (заданная как
поле `instruction:` артефакта в форкнутой схеме) + внешняя механическая проверка, которую запускает
**gate** проекта. У OpenSpec нет нативного примитива `governs:`/покрытия/теста-на-спецификацию/gate,
и именно поэтому зубы — внешние.

## Воркфлоу и жизненный цикл

Изменение движется через DAG артефактов OpenSpec:

```
proposal → specs → design → tasks → apply → (judge)
```

- **proposal** — что меняется и почему; плюс **Affected surfaces & rollback** (design использует
  поверхности, чтобы извлечь стандарты, чей `governs:` их покрывает).
- **specs** — авторство функциональных capabilities И стандартов. Именно здесь ЖИВУТ стандарты. Если
  цикл design пометил CREATE или REVISE, спецификация стандарта / delta пишется *здесь*.
- **design** — технический дизайн + **цикл стандартов** (ниже). design РЕШАЕТ по стандартам;
  specs ИХ ПИШЕТ — писать стандарт в design значило бы перевернуть #0 (производное, записанное
  решение, опережает свою исходную спецификацию).
- **tasks** — насаждающий механизм @ tier + тест-на-спецификацию + миграция + покрытие + регрессия + непрерывный
  gate. **Дисциплина SOURCE-FIRST (#0):** чтобы изменить поведение, СНАЧАЛА правьте спецификацию/сценарий (как
  delta), затем перепроецируйте его тест и код — никогда не редактируйте производное за спиной сценария,
  которого вы не трогали.
- **apply** — реализация tasks (унаследовано из форка; отслеживает `tasks.md`). Код рождается здесь.
- **judge** — выполняется ПОСЛЕ apply (он судит реализованный diff). Опциональный артефакт; перезапускаемый.

Полный git-жизненный цикл (из `templates/governance.md`):

```
0. (brownfield) задан режим adoption — см. Brownfield adoption.
1. START   — openspec new change <id>; ветка: git worktree add ../<id> -b change/<id>
2. AUTHOR  — proposal → specs → design → tasks. design РЕШАЕТ по стандартам; specs ИХ ПИШЕТ.
3. BUILD   — реализуйте tasks; запускайте GATE после каждой группы задач (непрерывно) до зелёного,
             разрешая/обосновывая каждого жёстко-локального подозреваемого judge.
4. ARCHIVE — openspec archive <id> → сливает delta-спецификации в openspec/specs/, перемещает изменение
             в changes/archive/. Перезапустите gate на обновлённых спецификациях до зелёного.
5. PR      — откройте PR из change/<id>. CI запускает gate (мягкий judge постит подозреваемых).
6. REVIEW  — человек одобряет, ратифицируя любые CREATE/REVISE стандартов (и спецификацию как источник правды).
7. MERGE   — squash-merge в дефолтную ветку. Прямые push заблокированы на хосте.
```

### Цикл стандартов

Перед написанием дизайна и всякий раз, когда возникает сквозное решение:

- **RETRIEVE** — вытяните стандарты, чей `governs:` соответствует поверхностям, которых касается это
  изменение (из proposal). Примените их. Извлечение — это эффективность с первой попытки; gate — это
  страховка на случай промахов.

Затем для каждого сквозного решения:

- **COMPLY** — укладывается в существующий стандарт → следуйте ему.
- **CREATE** — попадает на неуправляемую почву → ОПРЕДЕЛИТЕ потребность в новом стандарте и задайте его
  requirement / насаждающий механизм / tier / test (и, если архитектурный, его Decision Record). design РЕШАЕТ;
  specs ПИШЕТ спецификацию стандарта. Склоняйтесь к НЕМНОГИМ — стандарт облагает налогом каждое будущее изменение.
- **REVISE** — в напряжении с существующим стандартом → решите по DELTA (что ослабляет / ужесточает /
  делает исключение) и её обоснование; specs пишет delta и дописывает Rationale log стандарта
  (v2). Честно различайте comply-против-revise; ослабление требует сильного обоснования.

**RESOLVE-THEN-AUTHOR.** Для каждой клаузулы ЗАФИКСИРУЙТЕ, будут ли её проверка и тест ПЕРЕИСПОЛЬЗОВАНЫ (существующая
проверка/тест проекта уже их покрывает) или написаны заново. Сначала разрешайте против существующих
проверок проекта; пишите заново лишь когда ни одна не подходит.

**RATIFICATION.** Каждый CREATE / REVISE выводится на поверхность для подписи человеком на ревью PR: мягкий CI-judge
постит подозреваемых «standards-changed», а человек одобряет PR. Никогда не само-сливайте изменение
стандартов — стандарты связывают весь проект.

## Gate

spec-first ссылается на «**gate** проекта» абстрактно, чтобы схема оставалась независимой от стека.
Gate **ВНЕШНИЙ** по отношению к OpenSpec — `openspec validate --strict` лишь один из его входов.
Это ЕДИНАЯ команда, которую запускают и CI, и локальный цикл авторства: **непрерывно во время
изменения (после каждой группы задач), а не только в конце**, и обязательно зелёная как ПОСЛЕДНЯЯ группа задач
перед завершением изменения.

Он выполняет **шесть шагов**, падает на первом красном для шагов 1–5, и запускает judge последним:

1. **Механические проверки проекта** — `<lint/type/test command>`. ПОЛНЫЙ существующий набор, а не только
   проверки этого изменения. (Насаждающие механизмы стандартов управления тоже выполняются здесь, как любой
   стандарт — без отдельного шага.)
2. **Целостность org-стандартов** — `<integrity command>` (пропустите, если не потребляете org-бандл). Каждая
   спецификация под `specs/_org/` ОБЯЗАНА совпадать с зафиксированным бандлом org-стандартов по его **НОРМАЛИЗОВАННЫМ**
   полям, несущим правила (frontmatter `test`/`tier`/`governs` + набор requirement/scenario,
   игнорируя пробелы/комментарии — сырое побайтовое сравнение слетает при переформатировании).
   Локально ослабленный org-стандарт — это запрещённое расхождение — падение.
3. **Покрытие** — `<coverage command>`. Для каждой спецификации, ЗАТРОНУТОЙ ЭТИМ ИЗМЕНЕНИЕМ (добавлена/изменена
   delta): её `test:` ОБЯЗАН разрешаться в реальный, подключённый тест, И каждый `#### Scenario:` в ней ОБЯЗАН
   иметь ≥1 тест, несущий маркер этого сценария `@spec <capability>/<requirement>/<scenario>` (покрытие ключуется
   ТОЛЬКО на `test:` — нет `check:`, который нужно разрешать). Сценарий без проецирующего теста ⇒
   объявлено-но-не-спроецировано ⇒ падение. Gate
   также утверждает, что объявленный `tier` совпадает с тем, во что разрешается `test:` (машиночекаемый тест ⇒
   mechanical; зарегистрированный прогон judge ⇒ judge), и ПАДАЕТ при несовпадении. Judge-клаузула покрыта её
   зарегистрированной judge-проверкой, а не механическим тестом. Незатронутые спецификации повторно не аудируются;
   спецификации `adoption: legacy` освобождены до первого изменения. **Переприменение-после-CREATE — это под-проверка
   gate:** после первого (CREATE) archive новой спецификации gate ПАДАЕТ, если собранная спецификация лишена своего
   CORE (`test:`), так что забытое переприменение не может молча пройти покрытие.
4. **Diff регрессии** — `<regression command>`. Для каждой затронутой основной спецификации сравните её
   набор requirement+scenario с её BASELINE = **полной собранной спецификацией на merge-base этого изменения**
   (восстановленной из git, а НЕ из delta archive). Требование или сценарий, который
   исчез без явной delta `## REMOVED Requirements` (или `## RENAMED Requirements`),
   — молчаливая регрессия — падение.
5. **`openspec validate --strict`** — выход 0. (Структурные правила — ЖЁСТКИЕ ошибки даже без
   `--strict`; `## Purpose` < 50 символов падает только под `--strict` — см. факты 1.4.1 ниже.)
6. **Judge** (tier: judge-клаузулы) — `<judge command>`: агент в режиме print, рецензирующий
   управляемую область на ту часть, которую не покрывает ни одна механическая проверка.
   - **Локально:** БЛОКИРУЕТ. Не закрывайте задачу, пока у judge есть `open`-подозреваемый — исправьте его,
     или пометьте `justified` со ссылкой на Rationale log спецификации.
   - **В CI:** МЯГКО. Постите подозреваемых как аннотации PR / флаг «human-review-needed». НИКОГДА
     не валите сборку автоматически на недетерминированном вердикте.

**Зелёная-как-последняя-группа.** Gate выполняется на всём протяжении изменения, чтобы поломки всплывали рано;
ПОСЛЕДНЯЯ группа задач — это «полный gate зелёный», и она ОБЯЗАНА быть зелёной до завершения изменения.

Единственное, чего gate НЕ может — это заблокировать прямые push в дефолтную ветку — это
branch protection вашего git-хоста (см. Управление). Что gate доказывает, а что нет — это честная
заметка выше: согласованность + отсутствие-молчаливой-регрессии, а НЕ истинность спецификации.

## Тесты на каждой спецификации, покрытие, регрессия

**Тест на каждой спецификации.** `test:` — ЕДИНСТВЕННАЯ ссылка на насаждение, единственное поле frontmatter,
которое разрешает проверка покрытия. Тест — это ПРОЕКЦИЯ сценария, перепроецируемая из него; ФОРМА проекции
— выбор стека. Ни одна спецификация не уезжает без его разрешения:

- **Функциональная** → каждый сценарий проецируется в тест (unit / e2e / property / contract / type-level /
  рантайм-ассерт — что бы ни проверяло ТОТ сценарий).
- **Стандарт @ mechanical** → тест, утверждающий, что насаждающий механизм (unit/contract/type/runtime-тест ИЛИ
  lint/AST-правило — выбор стека, названный в теле теста и в прозе спецификации) ПОМЕЧАЕТ пример
  `violates` и ПРОПУСКАЕТ пример `conforms` (эти сценарии И ЕСТЬ фикстуры).
- **Стандарт @ judge** → зарегистрированная, перезапускаемая judge-проверка И ЕСТЬ её тест.

**Покрытие** аудирует ИЗМЕНЕНИЕ, а не всё ранее существовавшее дерево. На каждую затронутую спецификацию
(добавлена/изменена) КАЖДЫЙ `#### Scenario:` обязан иметь ≥1 тест, несущий его маркер `@spec
<capability>/<requirement>/<scenario>`; сценарий без проецирующего теста — это дефект и ОБЯЗАН
завалить. Это строже прежней привязки на уровне требования и есть точная операционализация принципа
«тесты — это проекции сценариев». (Judge-клаузула считается покрытой её зарегистрированной judge-проверкой;
покрытие НЕ требует механического теста judge-клаузулы. Спецификации `adoption: legacy` освобождены до
первого изменения. Отдельный `<coverage --all>` — это неблокирующий backlog-отчёт, который может
сканировать всё.)

Добавьте **маркер трассируемости**, привязывающий каждый тест к СЦЕНАРИЮ, который он проецирует — по
умолчанию `@spec <capability>/<requirement>/<scenario>` как комментарий в файле теста — чтобы проверка
покрытия могла его сгрепать. Один тест МОЖЕТ нести НЕСКОЛЬКО маркеров `@spec`, по одному на каждый
проецируемый им сценарий; маркер сидит на каждом тестовом CASE (на каждом блоке `it()`/`test`, или на
каждой группе фикстур RuleTester `valid`|`invalid`). Check-тест `standard@mechanical`, покрывающий
`conforms` И `violates`, несёт ДВА маркера. Имена сценариев сопоставляются как имена требований (с учётом
регистра, с нормализацией завершающих пробелов); переименование сценария ⇒ перепроецируйте его маркер.

**Регрессия** запускает ПОЛНЫЙ существующий набор проверок/тестов (изменение, которое зеленит свои проверки, но
краснит существующую, — это регрессия и падает), И сравнивает набор requirement+scenario каждой затронутой
основной спецификации с её **BASELINE = собранной спецификацией на merge-base**
(`openspec/specs/<cap>/spec.md` в том виде, в каком она была до применения этих delta — **восстановленной из git,
а НЕ из delta archive**, которая лишь diff ADDED/MODIFIED/REMOVED и не может реконструировать
прежний набор). Требование или сценарий, который исчез без явной delta `## REMOVED
Requirements`, — это молчаливая регрессия (MODIFIED archive заменяет ВЕСЬ блок требования) — падение. Затронутая
спецификация БЕЗ собранной формы на merge-base (только что добавленная, или первое изменение спецификации
`adoption: legacy`) не имеет baseline — ею вместо этого управляет покрытие, а baseline регрессии
устанавливается начиная с этого изменения.

### Эмпирические факты OpenSpec 1.4.1

Это протестированные, чувствительные к версии поведения — перепроверяйте при апгрейде:

- **Формат delta-спецификаций (HARD).** Delta-спецификации изменений используют уровень-2 `## ADDED|MODIFIED|REMOVED
  Requirements`, уровень-3 `### Requirement: <name>`, уровень-4 `#### Scenario: <name>` (РОВНО
  четыре решётки). Тело требования ОБЯЗАНО содержать литеральный `SHALL` или `MUST`. Собранная основная спецификация
  нуждается в `## Purpose` + `## Requirements` с ≥1 таким требованием + ≥1 сценарием. Переименование этих
  токенов проходит `schema validate`, но молча ломает `validate`/`archive`.
- **`validate --strict`.** `## Purpose` < 50 символов падает ТОЛЬКО под `--strict`. Структурные
  правила (Purpose + Requirements присутствуют; каждое требование содержит литеральные SHALL/MUST + ≥1 сценарий) — это
  ЖЁСТКИЕ ошибки, которые падают даже без `--strict`. `## Why` < 50 символов и > 10 delta НЕ валят
  strict — если вам нужны эти планки, добавьте их как свой собственный grep/length-подшаг, а не как заявление
  `--strict`.
- **Archive пересобирает файл целиком (устойчивое состояние MODIFY).** При archive MODIFY против существующей
  собранной спецификации `openspec archive` сохраняет всё ВЫШЕ `## Requirements` и любую завершающую секцию `## `,
  и УДАЛЯЕТ всё внутри `## Requirements`, что не является блоком `### Requirement:` —
  поэтому пользовательский frontmatter, ADR и rationale log живут ВЫШЕ `## Requirements` или в
  завершающей секции, никогда между требованиями. ПЕРВЫЙ/CREATE archive — исключение (следующий
  пункт): он собирает из тела ADDED delta В ОДИНОЧКУ, перенося его дословно и отбрасывая CORE/ADR/
  rationale — поэтому держите CREATE-delta минимальными и переприменяйте после.
- **Archive-при-CREATE отбрасывает не-requirement-контент (оговорка про переприменение).** ПЕРВЫЙ
  archive спецификации (CREATE) пересобирает основную спецификацию из ADDED delta В ОДИНОЧКУ — он ОТБРАСЫВАЕТ frontmatter
  CORE, ADR и rationale log, и подставляет заглушку `## Purpose` как «TBD». Лишь последующие MODIFY-
  archive сохраняют этот контент. **Поэтому после первого archive новой спецификации ПЕРЕПРИМЕНИТЕ её
  frontmatter CORE (`test:`) + ADR + rationale log в `openspec/specs/<name>/spec.md` и заново заполните Purpose,
  затем перезапустите gate** — без CORE покрытию нечего читать. Это переприменение ПОДСТРАХОВАНО gate'ом
  (CREATE, приземлившийся без своего CORE, ПАДАЕТ на покрытии; см. шаг 3), так что забытое переприменение
  не может молча пройти. (Блок tasks несёт обращённый-к-человеку шаг.)
- **Archive сопоставляет имена требований С УЧЁТОМ РЕГИСТРА**, нормализуя только завершающие пробелы —
  держите регистр маркера трассируемости точным, и привяжите матчер регрессии к той же
  нормализации только-завершающих-пробелов с учётом регистра.

### Механизм доставки openspec-instructions

OpenSpec (проверено на 1.4.1) выдаёт, на каждый артефакт, блок `<template>` с преамбулой «Use this as the
structure for your output file. Fill in the sections» И отдельный блок `<instruction>` («AI instructions for
creating this artifact»). Поэтому проза how-to GUIDANCE (guidance specs «Capabilities & standards», guidance
design «Standards loop», guidance tasks «Enforcement tasks/coverage/regression/re-apply» и guidance proposal)
задаётся как поле `instruction:` артефакта в форкнутом `schema.yaml` — а НЕ дописывается в файл шаблона. Файлы
`template:` остаются чистыми СКАФФОЛДАМИ-ЗАПОЛНЯЛКАМИ; подлинная СТРУКТУРА документа (например, заголовок
proposal `## Affected surfaces & rollback`) МОЖЕТ остаться небольшим дополнением к шаблону.

`openspec update` пишет только **generic**-обёртки skill/command из установленной CLI — они НЕ несут вашей
guidance. Ваша guidance связывает агента во **время авторства**: обёртки вызывают `openspec instructions
<artifact> --change <id>`, который выдаёт оба блока (`instruction:` = ваша guidance, `template:` = ваш
скаффолд). Так что логика живёт в схеме и доставляется по требованию, а не запекается в skills. Дымовой тест:
`openspec new change probe && openspec instructions specs --change probe` должен напечатать guidance
«Capabilities & standards». Если нет, guidance не подключена, и схема инертна.

## Управление — сделать воркфлоу единственным путём

Хотите, чтобы «каждое изменение проходит через OpenSpec» было *правилом*, а не надеждой? Это просто ещё стандарты.
`templates/governance.md` поставляет три затравочных стандарта (они следуют `templates/standard.md`, ложатся
под `specs/governance/<name>/spec.md` или `specs/_org/`):

- **changes-via-openspec** — дефолтная ветка принимает только заархивированные изменения OpenSpec; каждый
  несгенерированный файл в diff трассируется к tasks/specs этого изменения.
- **branch-per-change** — ровно одно изменение на ветку/worktree, именованную `change/<id>` (id ==
  суффикс ветки).
- **change-lifecycle** — gate-зелёный + одобрено человеком + заархивировано до слияния; одобрение
  РАТИФИЦИРУЕТ спецификацию как источник правды.

**Разделение in-repo-против-host.** Управление — это две половины, и обе обязательны:

- **В репозитории (под контролем версий):** три насаждающих механизма управления выше (каждый названный в
  теле `test:` своего стандарта), запускаемые gate (шаг 1). Они отталкиваются от `changes/`, имени ветки и
  diff PR. Только механические части проверяемы в репозитории: archive-присутствует, единственное-изменение,
  совпадение-имени.
- **На git-хосте (не может жить в репозитории):** **branch protection** на дефолтной ветке —
  нет прямых push, требуется PR, требуется зелёная CI-проверка gate, требуется ≥1 одобрение. Это
  та половина, которая делает «все изменения через воркфлоу» связывающей. Без неё правило рекомендательное; с
  ней — связывающее. Запишите точные настройки хоста в `project.md`.

`changes-via-openspec` и `branch-per-change` идентичны между проектами — первейшие кандидаты
для бандла org-стандартов, чтобы каждый репозиторий наследовал одно и то же неизменяемое правило.

## Внедрение в brownfield

spec-first управляет **изменениями**, а не вашим ранее существовавшим деревом — поэтому он ставится зелёным на реальный репозиторий
и храпово поднимает покрытие по одному изменению за раз. Поле `adoption:` — это legacy-храповик.
Перед первым изменением:

- **Сузьте scope строго до изменённых путей** (или один раз приведите репозиторий к чистоте `validate --strict`), чтобы
  несвязанное изменение никогда не краснело на legacy-благообразности.
- **Пометьте каждую ранее существовавшую спецификацию `adoption: legacy`** — `legacy` освобождает её от проверки покрытия
  до её следующего изменения, когда освобождение спадает («тронул — специфицируй»).
- Затем **enforce-on-touch**: каждое изменение полностью специфицирует и тестирует то, что добавляет или изменяет. Покрытие
  и `--strict` **привязаны к изменению**, никогда не аудит всего дерева.

**Затравка стандартов из существующих соглашений.** На свежем внедрении RETRIEVE ничего не находит, поэтому
каждое сквозное решение выглядит как неуправляемая почва для CREATE. Засейте корпус из того, что вы
УЖЕ насаждаете: для каждого существующего правила (конфиг eslint/biome, флаги tsconfig, проверки CI,
соглашения CONTRIBUTING) напишите СТАНДАРТ, чей `test:` РАЗРЕШАЕТСЯ в этот существующий инструмент (насаждающий
механизм, названный в его теле) — это resolve-then-author, документирующий насаждение, которое у вас уже есть,
а не новая машинерия. Пометьте каждый `adoption: legacy`, сузьте его `governs:` и склоняйтесь к НЕМНОГИМ. Это
даёт RETRIEVE на этапе design нечто реальное для нахождения и превращает неявные соглашения в насаждаемые,
рецензируемые стандарты по одному за раз.

(Greenfield: пропустите всё это — нет legacy-дерева, поэтому `adoption:` опускается везде.)

## Стандарты, общие для организации

> **Вторичная опциональная возможность.** Главное — что стандарты связывают во *всём
> проекте*; это бонус для организаций, желающих *один и тот же* стандарт насаждённым во многих репозиториях.

Переносимая единица — это **СПЕЦИФИКАЦИЯ стандарта** — его правило + СЦЕНАРИИ conforms/violates — которая
стек-агностична. Вы делитесь СЦЕНАРИЕМ; каждый потребляющий проект ПРОЕЦИРУЕТ эти сценарии в СВОЮ
СОБСТВЕННУЮ форму насаждения (свой собственный тест). Переиспользование — это **copy-in seed**, той же
формы, что и сам бандл схемы (у OpenSpec нет ссылок по remote-URL — [issue
#1131](https://github.com/Fission-AI/OpenSpec/issues/1131)):

- **Поставляйте** версионированный бандл `org-standards/`: полные экземпляры `standard.md`, каждый
  со scope `governs:`, каждый несёт своё правило + сценарии conforms/violates. (Общий конфиг, который вы
  тоже поставляете — например, базовый eslint-config / tsconfig — это удобная ЦЕЛЬ ПРОЕКЦИИ, а не общая
  единица.)
- **Внедряйте** в каждый проект копированием их под зарезервированный префикс `specs/_org/<name>/spec.md`,
  **фиксацией** версии бандла в `project.md` и ПРОЕЦИРОВАНИЕМ сценариев каждого стандарта в локальный тест
  в собственной форме насаждения проекта.
- **Делайте их обязательными** через шаг целостности org-стандартов gate (шаг 2 gate): каждая
  спецификация `specs/_org/` должна совпадать с зафиксированным бандлом по его **НОРМАЛИЗОВАННЫМ** полям, несущим правила
  (frontmatter + набор requirement/scenario, игнорируя пробелы/комментарии — сырое побайтовое сравнение
  слетает при переформатировании), так что локально ослабленное org-правило краснит gate. СПЕЦИФИКАЦИЯ
  (включая сценарии) общая и неизменяемая; проекция/тест — ЛОКАЛЬНЫ. `specs/_org/` никогда не является
  локальной целью delta, поэтому archive никогда его не переписывает. Чтобы изменить его, **отправьте REVISE вверх по течению
  и подвиньте фиксацию** — локально ослабить его нельзя.

> **Переиспользовать ПОТОК против переиспользовать ПРАВИЛА.** Положив `spec-first/` в `~/.local/share`, вы делитесь
> *пустым workflow*, а не вашими стандартами — стандарт это *экземпляр* спецификации, а не шаблон, поэтому он
> не может ехать в `schema.yaml`. Чтобы делиться насаждаемыми правилами, используйте библиотеку org-shared выше.

## Сквозной пример от начала до конца

`examples/add-rate-limit/` — это полное изменение, проведённое через весь воркфлоу от начала до конца. Оно
демонстрирует: **функциональную capability** (поведение rate-limiter, со сценариями Given/When/Then,
каждый из которых проецируется в тест); **CREATE стандарта**, пойманный циклом стандартов design на неуправляемой
почве (сквозная rate-limit-policy со scope `governs:`, ADR и
сценариями conforms/violates, которые служат заодно фикстурами теста); **frontmatter CORE** на
обеих формах; **tasks**, которые resolve-then-author каждый насаждающий механизм, тест и пер-сценарный
маркер `@spec <capability>/<requirement>/<scenario>`;
**gate**, запущенный непрерывно и зелёный как последняя группа; оговорку про **переприменение archive-при-CREATE**
для только что созданных спецификаций; и **ратификацию** — CREATE стандарта выводится на поверхность на ревью
PR (мягкий CI-judge постит подозреваемого «standards-changed»; это изменение не несёт judge-tier-
клаузулы, поэтому шаг judge выполняется и не находит ни одной). Читайте его рядом с этим README, чтобы увидеть каждую секцию
выше в одном конкретном изменении.

## Что внутри

```
.
├── README.md                       # этот файл
├── LICENSE
├── openspec/schemas/spec-first/
│   ├── schema.yaml                 # полная форма схемы: 4 базовых артефакта + опциональный `judge` + apply
│   ├── SETUP.md                    # пошаговая установка
│   └── templates/
│       ├── _additions.md           # guidance-проза → задаётся как `instruction:` каждого артефакта в schema.yaml (proposal/specs/design/tasks)
│       ├── standard.md             # полная archive-безопасная форма для спецификации СТАНДАРТА (+ ADR + rationale log)
│       ├── gate.md                 # блок gate из 6 шагов, встраиваемый project.md
│       ├── governance.md           # затравочные стандарты: changes-via-openspec, branch-per-change, change-lifecycle
│       ├── project.md              # контракт проекта: gate + org-фиксация + branch-protection + adoption
│       └── judge.md                # шаблон judge (жёстко локально, мягко в CI)
└── examples/
    └── add-rate-limit/             # полное изменение, проведённое через весь воркфлоу от начала до конца
```

`schema.yaml` объявляет четыре базовых артефакта (`proposal` → `specs` → `design` → `tasks`),
опциональный артефакт `judge` (`requires: [tasks]`, выполняется после apply) и унаследованный блок
`apply:`. `standard.md`, `gate.md`, `governance.md` и `project.md` — это **референсные шаблоны** —
накладываются при установке, без строки в `schema.yaml`; только `judge.md` получает строку.

## Установка

Это **оверлей**, а не схема с нуля: он строится на **форке встроенной схемы
`spec-driven`**, так что наследует формат delta-спецификаций (требования `ADDED` / `MODIFIED` / `REMOVED`),
на который опираются `openspec archive` и `openspec sync`. Вот почему установка начинается с
форка.

1. В вашем проекте: `openspec schema fork spec-driven spec-first` — пишет
   `openspec/schemas/spec-first/` (локально для проекта, под контролем версий).
2. Задайте guidance-блоки из `templates/_additions.md` как поля `instruction:` артефактов в вашем
   форкнутом `schema.yaml` (`proposal` / `specs` / `design` / `tasks`) — а НЕ дописывайте их в файлы
   шаблонов. Файлы `template:` остаются чистыми скаффолдами-заполнялками; добавьте только небольшую
   структуру `## Affected surfaces & rollback` в шаблон proposal. (Аддитивные пер-командные КОНСТРЕЙНТЫ —
   например, «proposal'ы этой команды обязаны включать план отката» — относятся к `config.rules`
   (пер-артефакт), отдельно от универсального воркфлоу в `instruction:`/шаблонах.)
3. Скопируйте `templates/judge.md`, `templates/standard.md`, `templates/gate.md`,
   `templates/governance.md` и `templates/project.md` в `templates/` схемы, и добавьте
   ТОЛЬКО строку `judge` в ваш форкнутый `schema.yaml` (сохраните остальные строки форка — они несут
   правильные указатели `template:` и блок `apply:`). `standard.md`, `gate.md`, `governance.md`
   и `project.md` — референсные шаблоны — без строки в `schema.yaml`. (Согласуйте каждый указатель `template:`
   с именем файла на диске — форк может выдать `spec.md`; несовпадение валит `schema validate`.)
4. `openspec schema validate spec-first`.
5. В `openspec/config.yaml` задайте `schema: spec-first` + `context: <your stack>`, затем `openspec
   update`. **Примечание:** `openspec update` пишет только *generic*-обёртки skill/command — он НЕ
   запекает в них вашу guidance. Эта guidance связывает агента во время авторства
   через `openspec instructions <artifact> --change <id>` (блок `<instruction>` per-artifact),
   который вызывают эти обёртки. **Дымовой тест:** `openspec new change probe && openspec instructions
   specs --change probe` должен напечатать guidance «Capabilities & standards».

Затем **определите gate** (обязательно — это то, что заставляет воркфлоу кусаться): вставьте `templates/gate.md`
в `openspec/project.md` и подключите его единственную команду (6 шагов). И, рекомендуется, **внедрите
управление** + задайте **branch protection** на вашем хосте. Полные детали в
`openspec/schemas/spec-first/SETUP.md`.

## Контракт project.md

Всё специфичное для стека живёт в `openspec/project.md` — ЕДИНСТВЕННОМ файле, где проект встречается с
воркфлоу. Он несёт **четыре вещи**, на которые шаблоны схемы ссылаются абстрактно:

1. **Gate** — команда, 6 проверок и маркер трассируемости (вставьте из `gate.md`).
2. **Фиксация org-стандартов** — бандл + версия, с которыми шаг целостности сопоставляет `specs/_org/`,
   или `none`.
3. **Настройки branch-protection** — правила хоста, которые gate не может насадить (нет прямых push,
   требуется PR, требуется зелёный gate, ≥1 одобрение). Записаны, чтобы гарантия была аудируемой.
4. **Режим adoption** — greenfield | brownfield, строгий scope и заметка про legacy-храповик.

Скопируйте `templates/project.md` и заполните слоты `<…>`; он встраивает `gate.md`. Шаблоны схемы
остаются независимыми от стека; этот один файл — место, где проект встречается с воркфлоу.

## Использование между проектами / в команде

- **Все ваши собственные проекты:** положите папку `spec-first/` в
  `~/.local/share/openspec/schemas/` (глобально для пользователя) — тогда она доступна везде, одна копия
  воркфлоу.
- **Команда:** этот репозиторий *и есть* дистрибутив. Коллеги клонируют/копируют папку к себе. У OpenSpec
  **нет ссылок на схему по remote-URL** (нельзя указать проекту на GitHub-схему по URL — см.
  [issue #1131](https://github.com/Fission-AI/OpenSpec/issues/1131)); бандл приносится локально,
  так же, как работает любая community-схема.

## Версионирование и dogfooding

Протестировано против **OpenSpec 1.4.1**; archive-безопасность и поведение `validate --strict`
чувствительны к версии — перепроверяйте при апгрейде. Апгрейды — ручной copy-in: сравните `templates/` и
перезапустите `openspec schema validate`. И честно: **этот репозиторий — это *бандл* схемы, а не сам по себе
spec-first-проект** — он поставляет воркфлоу, он не работает под ним (дистрибутив шаблонов не имеет
рантайма или набора тестов для gate).
